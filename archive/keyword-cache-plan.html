03
DB 테이블 설계
📦 keyword_cache 검색 키워드별 연관 키워드 + 검색량 캐시
🔑 id uuid PK, 자동 생성
source_keyword text 입력한 원본 키워드 (예: "자동차커버") UNIQUE
keywords jsonb [{keyword, volume}, ...] — 연관 키워드 + 검색량 배열
step1_count int Step 1 AI 생성 개수
step2_count int Step 2 이미지 검증 통과 개수
total_keywords int 최종 키워드 수
volume_updated_at timestamptz 검색량 마지막 갱신일 INDEX
created_at timestamptz 최초 생성일
hit_count int 캐시 히트 횟수 (인기도 추적)
📋 keyword_search_log 검색 이력 전체 로그 (누가, 언제, 뭘 검색했는지)
🔑 id bigint PK, 자동 증가
source_keyword text 검색한 키워드
mode text 'single' | 'excel' — 어느 탭에서 검색했는지
cache_hit boolean 캐시에서 반환했는지 여부
result_count int 반환된 키워드 수
response_ms int 응답 시간 (밀리초)
searched_at timestamptz 검색 시각 INDEX
04
자동 갱신 크론 설계
네이버 검색량 업데이트 주기: 네이버 검색광고 API는 월간 검색량을 제공하며, 매월 초(1~5일 사이) 전월 데이터로 갱신됩니다. 정확한 갱신일은 공식적으로 명시되지 않으므로, 매월 7일에 갱신 크론을 돌려 확실하게 최신 데이터를 수집합니다.
🔄 월간 검색량 갱신
0 3 7 * *
매월 7일 새벽 3시 실행
keyword_cache의 전체 키워드에 대해
네이버 API로 최신 월간 검색량 재조회
→ keywords jsonb 내 volume 값 업데이트
→ volume_updated_at 갱신
🧹 오래된 캐시 정리
0 4 1 * *
매월 1일 새벽 4시 실행
90일간 hit_count = 0인 키워드 삭제
(아무도 안 쓰는 키워드 정리)
→ DB 크기 관리
→ 활성 키워드만 유지
05
Edge Function 변경 로직
⚡ keyword-optimizer (v2) 캐시 우선 조회 로직 추가 
// 1. 요청 수신
POST { productName, imageUrl, mode }

// 2. 캐시 조회
SELECT * FROM keyword_cache
WHERE source_keyword = productName
AND volume_updated_at > now() - interval '30 days'

// 3-A. 캐시 히트 → 즉시 반환
IF (cached) {
  UPDATE hit_count += 1
  INSERT INTO keyword_search_log (cache_hit: true)
  RETURN cached.keywords  ← ~0.3초
}

// 3-B. 캐시 미스 → 기존 3-Step 실행
Step 1: Gemini AI 키워드 생성
Step 2: Vision AI 이미지 검증
Step 3: 네이버 API 검색량 조회

// 4. 결과 저장
UPSERT INTO keyword_cache (...)
INSERT INTO keyword_search_log (cache_hit: false)

RETURN keywords  ← ~8초
06
데이터 성장 시나리오
📈 축적 예상 사용할수록 빨라지는 구조
1개월 ~200건 초기 키워드 수집. 캐시 히트율 ~20%
3개월 ~800건 주요 상품 키워드 대부분 캐시. 히트율 ~60%
6개월 ~2,000건 덕우 전 상품 커버. 히트율 ~80%
1년 ~5,000건 시장 키워드 맵 완성. 히트율 90%+. 대부분 즉시 응답
핵심 가치: 사용하면 할수록 데이터가 쌓이고, 쌓일수록 빨라지고, 빨라질수록 더 많이 사용하게 되는 선순환 구조입니다. 1년 뒤에는 네이버 키워드 검색량 DB 자체가 자산이 됩니다.
